AWS Communication Protocol Interface

PROTOCOL GENERAL DESCRIPTION

The protocol disciplines two kind of data exchange transactions:

[1] Command and answer transaction:

  • AWS Command Frame -> MainCpu
    • MainCpu Answer frame -> AWS

[2] MainCpu status change transaction:

  • MainCpu Status Change Frame -> AWS
  • No answer from AWS is needed!

In the transaction [1] the AWS sends commands to be executed from the MainCpu;
the MainCpu shall acknowledge the command with proper answer frames (further described).

In the transaction [2] the MainCpu sends Status frames used to notify the AWS
about the internal status. Those frames shall not be knowledged by the AWS program.

The Command frame of [1] contains an ID code that is reported back in the MainCpu Answer frame:
this code can be either a progressive code or a unique code.
In the Command/Answer scheme of [1], MainCpu can sends back several possible answer frame types:

  • <Immediate OK> frame: the receiver processed with success the requested command;
  • <Immediate NOK> frame: the processed command fails or cannot be executed;
  • <Delayed Command> frame: the MainCpu accepts the command but the command takes time to complete;
    • <Delay Completed> Frame: the MainCpu shall send back this frame to notify the command completion;
  • Command not implemented (or NA) frame: the requested command is not supported or not implemented;

COMMUNICATION CHANNEL

The communication takes place over the Tcp/Ip protocol:

  • The MainCpu is the server of the communication;
  • The IP address and port are 127.0.0.1@10001 if the AWS software resides into the MainCpu PC;
  • If the AWS software is outside of the MainCpu PC, then the IP and port shall be set according with the Network used.
    Attention
    The IP address and port are stored into the system.ini configuration file,
    and can be changed if necessary to accomodate a specific installation.

PROTOCOL DATA FRAME FORMAT

DATA TYPE AND ENCODING FORMAT

The frames are composed by characters formatted with the UTF-16-LE;
All the parameter fields shall be converted from its string representation
to the target type, based on the specific command implementation.

STRUCTURE OF DATA FRAME

The data frame formats are of four possible forms: 

   [1]  <ID FRAME_LEN % CMD PAR1 .. PAR-N %>            AWS command frame;
   [2]  <ID FRAME_LEN % "OK/NOK" CODE OPT1 .. OPT-N %>  MainCpu answer frame;
   [3]  <'A' FRAME_LEN % ID CODE OPT1 .. OPT-N %>       MainCpu delayed completed command answer;
   [4]  <'S' FRAME_LEN % CMD PAR1 .. PAR-N  %>          MainCpu Status command to AWS

    Where:
    '<' and '>  : delimitators of the data frame
    '%'         : delimitators of the frame content;
    ID          : command idntification code
    'A'         : key character identifying the MainCpu <Delay Completed Frame> answer frame format;
    'S'         : key character identifying the MainCpu <Status> command frame format;
    FRAME_LEN   : integer number in string format, equals to the total number of characters composing the frame;
    PAR1 - PAR-N: list of command parameters, in string format, separated by spaces;
    OPT1 - OPT-N: list of optionnal parameters, in string format, separated by spaces;

<COMMAND> DATA FRAME FORMAT [1]

The Command data frame frame format is based on the form [1] as following described: 

    <ID FRAME_LEN % COMMAND PARAM-1 PARAM-2 .... PARAM-N %>
    Where:
    COMMAND: is a String defining a command code;
    PARAM-1, .. , PARAM-N: are a list of parameters, in string format, of variant type. The actual type depends by the command implementation.

<IMMEDIATE OK> DATA FRAME FORMAT [2]

The <immediate ok> data frame format is based on the form [2];
It shall be sent back by the MainCpu if the requested command has been successfully processed.

    <ID FRAME_LEN % OK 0 OPT1 ... OPT-N %>
    Where:
    ID: is the ID of the processed Command Frame
    OPT1 .. OPT-N: are optionals data sent back to the Sender
    The " OK 0 " part of the frame identifies the frame type.

<DELAYED OK> DATA FRAME FORMAT [2]

The <delayed ok> data frame format is based on the form [2];
It shall be sent back by the MainCpu if the command takes time to completes.

    <ID FRAME_LEN % OK TIME OPT1 ... OPT-N %>
    Where:
    ID: is the ID of the processed Command Frame;
    TIME: this is a number defining approximatelly a waiting time.
    OPT1 .. OPT-N: are optional data sent back in string format.

As soon as the command terminates, the MainCpu shall send back to the AWS
the <Delay Completed Frame> data format [3].

    <A FRAME_LEN % ID REASULT OPT1 .. OPT-N %>
    Where:
    ID: is the ID of the processed Command Frame
    REASULT:
        - 0: if the command is successfully terminated;
        - CODE: error code;
    OPT1 .. OPT-N: are optional data sent back in string format.

<IMMEDIATE NOK> DATA FRAME FORMAT [2]

The <immediate NOK> data frame format is based on the form [2];
It shall be sent back by the MainCpu in the case the command fails or
cannot be executed.

The format is the following:

    <ID FRAME_LEN % NOK ERRCODE OPT1 ... OPT-N %>
    Where:
    ID: is the ID of the processed Command Frame;
    ERRCODE: is the failure code;
    OPT1 .. OPT-N: are optional data sent back in string format.

<NA> DATA FRAME FORMAT [2]

The <NA> data frame format is based on the form [2];
It shall be sent back by the MainCpu in the case
the requested command should not be implemented or not supported:

    <ID FRAME_LEN % NA %>

<STATUS> DATA FRAME FORMAT [4]

The <STATUS> data frame format is based on the form [4];
It shall be sent by the MainCpu in order to notify the AWS of a change of internal status.

The frame format is the following:

    <'S' FRAME_LEN % CMD OPT1 .. OPT-N  %>
    Where:
    'S': is the status frame identifier character;
    CMD: is the notify command;
    OPT1 .. OPT-N: is a list of optional variable status.
Attention
The AWS shall never acknowledge a status frame.

AWS COMMAND DESCRIPTION

In the command frame [1], the first field represents the command code.

The Command code shall apply to the following rules:

  • The command strings shall start with:
    • "Get_": for commands getting data;
    • "Set_": for commands setting data;
    • "Exec_": for command executing functions;
  • The following strings are in any case reserved:
    • "OK"
    • "NOK"
    • "NA"

Example of valid commands are:

  • Set_ArmActivation;
  • Set_StartProcedure;
  • Get_CoompressorPaddle;
  • Get_CompressorThickness;
  • Exec_OpenStudy;

For the list of available commands, see Aws Command frames interface description;

STATUS COMMAND DESCRIPTION

The status command shall be preposed with the "STATUS_" string.

For the list of available status commands, see Status frames interface description.;

PARAMETER FORMAT SPECIAL CONSIDERATIONS

The protocol exchanges commands as well parameters in string format.

Data of Integer, Float, String type can be easily exchanged converting it in its string representation.

Because of data and command items are separated by spaces (the number of spaces doesn't matter)
if the Sender should send a composed string as a unique parameter, it shall delimit it into the " " characters.

Example: 

    <10 0051 % Exec_OpenStudy "Pico della Mirandola" %>
    Decoded items:
        command = OpenStudy;
        param1 = "Pico della Mirandola"


    <10 0049 % Exec_OpenStudy Pico della Mirandola %>
    Decoded items:
        command = OpenStudy;
        param-1 = "Pico";
        param-2 = "della";
        param-3 = "Mirandola"
